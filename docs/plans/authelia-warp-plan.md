# Implementation Plan: Authelia RBAC + WARP Enrollment

> Generated by `/PACT:plan-mode` on 2026-03-19
> Status: PENDING APPROVAL

## Summary

Deploy Authelia on VPS as Layer 2 RBAC (OIDC provider + forward-auth gateway) behind the existing Cloudflare tunnel. Caddy expands from public-only to also serve as the internal reverse proxy for all tunneled services, with Authelia forward-auth middleware. WARP client enrollment enables silent Cloudflare Access verification. Per-service migration with instant rollback at each step.

## Architecture

```
                    CLOUDFLARE EDGE
                         │
              ┌──────────┴──────────┐
              │  CF Access (L1)     │  ← WARP bypasses login prompt
              │  CF Tunnel helix-vps│
              └──────────┬──────────┘
                         │
                    cloudflared
                         │
           ALL HTTP → 127.0.0.1:8443
           SSH      → localhost:2222 (direct, unchanged)
                         │
              ┌──────────┴──────────┐
              │   Caddy (single)    │
              │                     │
              │  PUBLIC 0.0.0.0:443 │  track, rp (unchanged)
              │  INTERNAL :8443     │  all tunneled services
              └──────────┬──────────┘
                         │
         ┌───────────────┼───────────────┐
         │               │               │
    Forward-Auth    OIDC Native      Bypass
    (non-OIDC)      (native SSO)    (API/public)
         │               │               │
    ┌────┴────┐          │          ┌────┴────┐
    │Authelia │          │          │ s3 API  │
    │ :9091   │          │          │ (S3 key │
    │ (L2)    │          │          │  auth)  │
    └────┬────┘          │          └─────────┘
         │               │
   ┌─────┴─────┐   ┌────┴─────┐
   │Vaultwarden│   │ Harbor   │
   │OpenBao    │   │ MinIO    │
   │Postal     │   │ Grafana  │
   └───────────┘   │ Devtron  │
                   └──────────┘
```

**Three auth patterns:**
- **Forward-auth** (Vaultwarden, OpenBao, Postal): Caddy sends subrequest to Authelia before proxying. No OIDC support in these apps.
- **OIDC native** (Harbor, MinIO, Grafana, Devtron): App redirects to Authelia for SSO login. Caddy just proxies — no forward-auth needed.
- **Bypass** (s3.helixstax.net): S3 API uses access key auth. Incompatible with browser-based auth.

### Architecture Phase
**Effort**: Medium

#### Components

| Component | Change | Notes |
|-----------|--------|-------|
| Authelia | NEW | Docker container, port 9091 on localhost |
| Caddy | MODIFY | Add internal listener on 127.0.0.1:8443 with forward-auth |
| cloudflared | MODIFY | All HTTP services point to :8443 instead of direct ports |
| SSH | UNCHANGED | Stays ssh://localhost:2222 direct |
| Track/RP | UNCHANGED | Public Caddy on 0.0.0.0:443 |

#### Key Decisions (RESOLVED)

| Decision | Choice | Rationale | Consensus |
|----------|--------|-----------|-----------|
| Reverse proxy | Caddy (single instance) | Already installed, cleanest forward-auth syntax, no TLS needed internally | All 3 agree |
| Storage backend | SQLite | Single instance, low user count, zero dependencies | All 3 agree |
| Session backend | In-memory | Admin-only, re-login on restart acceptable. Add Redis when client-facing. | Cass + Ezra (Kit preferred Redis — overruled for simplicity at current scale) |
| User store | File-based YAML | One admin user, version-controllable, migrate to SQLite when team grows | Ezra recommended |
| S3 API auth | Bypass (no Authelia) | S3 Signature v4 incompatible with browser auth | Cass + Ezra agree |
| Fail mode | Fail-closed | Services inaccessible if Authelia down (safer than fail-open) | Cass + Ezra agree |
| 2FA | TOTP for secrets, one_factor for standard | CF Access already has MFA at L1, TOTP adds defense-in-depth for high-value targets | Ezra recommended |
| OIDC security | PKCE required, exact redirect URIs | Prevents authorization code interception and open redirect attacks | Ezra: non-negotiable |

#### Access Control Policy

| Service | Domain | Auth Pattern | Authelia Policy | 2FA |
|---------|--------|-------------|-----------------|-----|
| Authelia portal | auth.helixstax.net | Direct (it IS the auth) | bypass | N/A |
| Vaultwarden | vault.helixstax.net | Forward-auth | two_factor, group:admins | TOTP |
| OpenBao | bao.helixstax.net | Forward-auth | two_factor, group:admins | TOTP |
| Postal | postal.helixstax.net | Forward-auth | two_factor, group:admins | TOTP |
| Harbor | harbor.helixstax.net | OIDC native | bypass (app handles) | Via OIDC |
| MinIO Console | minio.helixstax.net | OIDC native | bypass (app handles) | Via OIDC |
| Grafana | grafana.helixstax.net | OIDC native | bypass (app handles) | Via OIDC |
| Devtron | devtron.helixstax.net | OIDC native | bypass (app handles) | Via OIDC |
| MinIO S3 API | s3.helixstax.net | S3 key auth | bypass | N/A |

#### Session Duration Alignment

| Layer | Secrets Services | Standard Services |
|-------|-----------------|-------------------|
| Cloudflare Access | 1h | 24h |
| Authelia session | 45min | 12h |
| Authelia inactivity | 15min | 2h |

Rule: Authelia session always shorter than Cloudflare Access session (Ezra: prevents stale L2 session outliving revoked L1).

#### Circular Dependency Warning (Ezra)

Authelia MUST NOT depend on OpenBao at startup. OpenBao is behind Authelia. If Authelia needs OpenBao secrets to start, and OpenBao needs Authelia to be accessible, neither can start.

**Solution**: Authelia's own config (JWT secret, session secret, OIDC client secrets) stays in files/env vars on the VPS, NOT in OpenBao. OpenBao stores secrets for other services that Authelia protects.

#### Implementation Sequence

**Phase 0: Pre-deployment hardening**
1. Rotate Cloudflare API key → scoped token
2. Verify service bindings (`ss -tlnp`)
3. Open emergency SSH port in Hetzner firewall (temporary)
4. Backup current configs (Caddyfile, cloudflared config)

**Phase 1: Deploy Authelia (no traffic yet)**
5. Create directory structure + config files
6. Generate admin password hash (argon2id)
7. Start Authelia container
8. Verify health: `curl http://localhost:9091/api/health`

**Phase 2: Expand Caddy (no traffic yet)**
9. Add internal listener block to Caddyfile
10. Validate + reload Caddy
11. Test internal routing: `curl -H "Host: vault.helixstax.net" http://127.0.0.1:8443/`

**Phase 3: Migrate services (per-service, ~2s interruption each)**
Migration order (lowest risk first):
1. auth.helixstax.net → Authelia portal
2. s3.helixstax.net → bypass (no auth change)
3. harbor.helixstax.net → OIDC native (test OIDC flow)
4. minio.helixstax.net → OIDC native
5. vault.helixstax.net → forward-auth (test forward-auth)
6. bao.helixstax.net → forward-auth
7. postal.helixstax.net → forward-auth

For each: update cloudflared config → restart cloudflared → verify

**Phase 4: OIDC client registration**
8. Configure Harbor OIDC client
9. Configure MinIO OIDC client
10. (Grafana/Devtron when deployed on K3s)

**Phase 5: Admin hardening**
11. Enroll TOTP for admin account
12. Close emergency SSH port
13. Test fail-closed behavior (stop Authelia, verify 502)

**Phase 6: WARP enrollment (parallel)**
14. Configure WARP device enrollment in CF Zero Trust
15. Install WARP on Windows
16. Enroll device
17. Set CF Access policy to require WARP
18. Verify silent authentication

#### Rollback at Each Phase

| Phase Failed | Rollback |
|-------------|----------|
| Phase 1 | `docker compose down` — no impact |
| Phase 2 | Restore Caddyfile.bak + reload — no impact |
| Phase 3 (per service) | Restore that service's cloudflared entry to direct port + restart |
| Phase 4 | Remove OIDC config from app — falls back to local auth |
| Phase 5 | N/A — hardening, no risk |
| Phase 6 | Remove WARP requirement from CF Access policy |

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Forward-auth bypass via proxy misconfig | Medium | Critical | Default-deny Caddy config, verification tests |
| OIDC redirect URI misconfiguration | Low (if exact) | Critical | Exact URIs, PKCE, no wildcards |
| Authelia SPOF | Low | High | Docker restart policy, break-glass procedure |
| Session conflict between CF + Authelia | Low | Medium | Different cookie names, aligned durations |
| Circular dependency (Authelia ↔ OpenBao) | High if not planned | Critical | File-based Authelia config, no OpenBao dependency |
| SSH lockout during migration | Low | Critical | Emergency SSH port open during migration |
| Global API key exposure | Medium | Critical | Rotate to scoped token BEFORE deployment |
