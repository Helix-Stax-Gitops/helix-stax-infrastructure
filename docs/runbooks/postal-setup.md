## 1. Architecture

### Email Flow

```
Apps / Mautic
     |
     | SMTP (587) or HTTP API
     v
  Postal
  (web, smtp, worker)
     |
     | SMTP (localhost relay)
     v
  Postfix (container)
     |
     | SMTP (587, STARTTLS)
     | with SASL credentials
     v
Amazon SES
     |
     | Outbound delivery
     v
  Recipient
```

### Domain Separation

| Domain | Purpose |
|---|---|
| `helixstax.com` | Public-facing brand. No outbound mail. Reply-to target. |
| `helixstax.net` | Transactional and internal mail. All Postal sending addresses live here. |

Postal manages `helixstax.net` delivery. SES handles the actual outbound hop. Replies go to `@helixstax.com` via Reply-To headers set per message.

## 4. Key Config Files

All paths are relative to `/opt/helix-stax/postal/` on the VPS unless otherwise noted.

### `docker-compose.yml`

Defines all six services. Key points:
- `postfix` service is built from `Dockerfile.postfix` in the same directory.
- `postal-net` is declared as `internal: true`.
- `helix-net` is declared as `external: true` (pre-existing shared network).
- Volumes mount `config/` into Postal containers and `postfix-main.cf` / `sasl_passwd` into the Postfix container.

### `config/postal.yml`

Main Postal application config. **Critical v3 format note** — see Troubleshooting section.

The smtp_relays entry must follow the URI string format under the `postal:` key:

```yaml
postal:
  smtp_relays:
    - "smtp://postfix:587"
```

Do not use the YAML object format (host/port fields). Postal v3 silently ignores the old format.

Other notable config sections:
- `web.host` — set to `postal.helixstax.net`
- `rails.secret_key_base` — pulled from OpenBao at deploy time
- `database.*` — MariaDB connection credentials (see Credentials section)
- `rabbitmq.url` — RabbitMQ connection string

### `Dockerfile.postfix`

Custom Alpine-based image. Installs:
- `postfix`
- `cyrus-sasl` and `cyrus-sasl-plain` (required for SMTP AUTH to SES)
- `ca-certificates`

Copies `postfix-main.cf` to `/etc/postfix/main.cf` and `sasl_passwd.db` into the image at build time.

**Important**: Postfix config is baked into the image at build time. Changes to `postfix-main.cf` require a rebuild (`docker compose build postfix && docker compose up -d postfix`). Runtime `postconf -e` commands do not persist across container restarts.

### `postfix-main.cf`

Postfix configuration for relaying to SES. Key directives:

```
relayhost = [email-smtp.us-east-1.amazonaws.com]:587
smtp_sasl_auth_enable = yes
smtp_sasl_password_maps = hash:/etc/postfix/sasl_passwd
smtp_sasl_security_options = noanonymous
smtp_tls_security_level = encrypt
smtp_tls_CAfile = /etc/ssl/certs/ca-certificates.crt
```

`smtp_tls_CAfile` is required. Without it, Postfix cannot verify SES's TLS certificate and connections time out silently. Alpine's CA bundle path differs from Debian — confirm the path matches what is present in the image.

### `sasl_passwd`

SMTP credentials file for SES relay authentication. Format:

```
[email-smtp.us-east-1.amazonaws.com]:587  <SES_SMTP_USER>:<SES_SMTP_PASSWORD>
```

After editing, regenerate the hash db:

```bash
postmap /etc/postfix/sasl_passwd
```

The `.db` file is what Postfix reads. Both files are baked into the Docker image — rebuild after any credential rotation.

Actual credentials are stored in OpenBao. See Credentials section.

## 5. Credentials

All secrets are stored in OpenBao (Vault-compatible). Do not store plaintext credentials in config files committed to version control.

| Secret | OpenBao Path |
|---|---|
| Postal database password | `secret/postal/database` |
| Postal secret_key_base | `secret/postal/app` |
| RabbitMQ credentials | `secret/postal/rabbitmq` |
| SES SMTP username | `secret/postal/ses` |
| SES SMTP password | `secret/postal/ses` |

To retrieve a secret for use during a rebuild or config update:

```bash
# Example — adjust path and field name
bao kv get -field=password secret/postal/database
```

Inject secrets into `postal.yml` and `sasl_passwd` at deploy time. Never commit files containing live credentials.

## 7. Port 25 and Bounce Handling

### Port 25 Block

Hetzner blocks inbound port 25 on VPS instances less than 30 days old. The postal VPS (5.78.145.30) was provisioned in mid-March 2026.

**Port 25 unblocked**: approximately April 17, 2026.

Until then:
- Outbound delivery works (via SES on port 587).
- Inbound SMTP (bounce processing, incoming mail) is blocked.
- Do not configure MX records on `helixstax.net` until port 25 is open and tested.

### Bounce Handling Workaround

While port 25 is blocked, use Amazon SES SNS notifications for bounce and complaint handling:

1. In the SES console, configure a SNS topic for bounce and complaint events on the verified identity.
2. Subscribe an HTTP/HTTPS endpoint (or Lambda) to the SNS topic to receive notifications.
3. Process bounce/complaint payloads and suppress addresses accordingly.

Once port 25 is unblocked, configure Postal to receive bounce email directly via the return-path domain (`rp.helixstax.net`).

### SES Sandbox

SES accounts start in sandbox mode. In sandbox:
- Only verified email addresses or domains can receive test mail.
- Sending limits apply.

Request production access through the SES console before going live with any bulk or transactional sending.

## 8. Troubleshooting and Lessons Learned

### smtp_relays format change in Postal v3

Postal v3 changed the `smtp_relays` configuration format. The v2 YAML object format (using `host:` and `port:` keys) is silently ignored in v3 — no error, no relay.

**Broken (v2 format — do not use):**
```yaml
postal:
  smtp_relays:
    - host: postfix
      port: 587
```

**Correct (v3 URI string format):**
```yaml
postal:
  smtp_relays:
    - "smtp://postfix:587"
```

If outbound mail appears to process in Postal but never reaches SES, check this first.

### Postal v3 has no authenticated SMTP relay support

Postal v3 cannot authenticate to a relay that requires SMTP AUTH (GitHub issue #2811). This is a known limitation with no current fix in the Postal codebase.

**Workaround**: Run Postfix as an intermediary container. Postal relays to Postfix unauthenticated (within the internal `postal-net` network), and Postfix handles SASL authentication to SES. This is the current production setup.

Do not attempt to configure Postal to authenticate directly to SES — it will not work.

### smtp_tls_CAfile required in Postfix for SES

Without `smtp_tls_CAfile` set to a valid CA bundle, Postfix cannot verify SES's TLS certificate. The connection times out or is refused with no helpful error in the log.

Ensure the path matches the CA bundle location inside the Alpine container (`/etc/ssl/certs/ca-certificates.crt`). Verify the file exists in the image:

```bash
docker compose exec postfix ls /etc/ssl/certs/ca-certificates.crt
```

### Postfix config is baked into the Docker image

The `Dockerfile.postfix` copies `postfix-main.cf` and `sasl_passwd.db` into the image at build time. Runtime changes via `postconf -e` do not persist.

Workflow for Postfix config changes:
1. Edit `postfix-main.cf` (and `sasl_passwd` if credentials changed).
2. If `sasl_passwd` changed, run `postmap sasl_passwd` to regenerate the `.db` file.
3. Rebuild the image: `docker compose build postfix`
4. Restart the container: `docker compose up -d postfix`

### Hetzner port 25 block on new VPS

Hetzner blocks port 25 on VPS instances for the first ~30 days. This affects inbound SMTP only — outbound via port 587 to SES works immediately. Plan deployments accordingly and do not configure inbound mail handling until the block is lifted.

### SES sandbox requires verified recipients

In SES sandbox mode, test mail can only be sent to email addresses or domains that have been verified in the SES console. Sending to an unverified address will fail silently or return a bounce. Request production access before any live sending.
